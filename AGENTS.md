# 项目开发强制准则

本文件适用于本仓库全部目录。任何AI或开发人员开始工作前，必须先阅读：

1. `README.md`
2. `docs/architecture.md`
3. `docs/development-workflow.md`
4. `docs/implementation-plan.md`

## 1. 单项原子开发

- 每次只允许处理计划表中的一个原子项，不得顺带实现后续模块。
- 开始前必须在计划表中将该项标记为“进行中”，并明确允许修改的路径。
- 发现相邻问题时只能记录为后续计划，不得越界修复。
- 当前项未完成、未验证、未推送前，不得开始下一项。

## 2. 设计先行

- 每个编码项必须有一个已完成并已推送的前置设计项。
- 详细设计统一放在`docs/design/`，文件名与计划编号一致。
- 设计必须包含目标、范围、非目标、接口、数据、流程、异常、测试、风险、回退和依赖。
- 设计完成后必须记录反思与检查：方案取舍、失败场景、架构一致性、范围检查和可验证性。
- 设计未通过检查或依赖不明确时，禁止编码。

## 3. 编码与验证

- 只能修改当前计划项列出的路径。
- 必须先运行与改动直接相关的单元测试和集成测试。
- 每个实现项必须提供并执行端到端验证，证明输入经过真实组件得到预期输出。
- AI只模拟输入；校验、存储、推理和查询结果不得由AI伪造。
- 完成后必须进行代码检视，至少检查正确性、范围、错误处理、安全、测试覆盖和无关改动。
- 测试或检视未通过时，不得提交为完成。

## 4. Git与计划回写

- 一个原子项对应一个主题明确的提交，不混入无关文件。
- 提交信息使用`docs|design|feat|fix|test(Pxx): 中文摘要`格式。
- 提交后必须主动推送当前分支到GitHub。
- 推送成功后，更新`docs/implementation-plan.md`中的状态、提交号和验证证据，再提交并推送计划更新。
- 只有实现提交和计划更新提交均推送成功，计划项才能标记为“已完成”。
- 禁止重写他人历史、强制推送、删除未知改动或跳过验证。

## 5. 停止并求助的条件

出现以下情况时停止当前项并向用户求助，不得擅自替换技术路线：

- 缺少设计指定的运行时、容器、模型、凭据或外部服务；
- 需要新增超出当前项范围的依赖或基础设施；
- 端到端验证无法在当前环境执行；
- GitHub认证失效或推送失败；
- 发现用户已有改动与当前项冲突；
- 需求变化会影响架构、接口或后续计划。

完整流程和检查清单见`docs/development-workflow.md`。
