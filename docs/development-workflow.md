# 原子开发与交付流程

## 1. 目的

本准则用于保证项目以可审查、可验证、可回退的方式逐步推进。每次工作只交付一个边界明确的原子成果，并在GitHub留下设计、实现、验证和进度记录。

## 2. 核心规则

1. 设计先于编码；
2. 设计项和实现项分开；
3. 一次只执行一个计划项；
4. 不实现当前范围之外的模块；
5. 每个实现必须经过端到端验证；
6. 设计需要反思和检查，代码需要测试和检视；
7. 每完成一个原子项必须提交并推送GitHub；
8. 推送后必须更新计划表进度；
9. 缺少必要环境或凭据时立即停止并求助。

## 3. 状态定义

| 状态 | 含义 |
| --- | --- |
| 待开始 | 尚未开始，前置条件可能未满足 |
| 进行中 | 当前唯一允许执行的计划项 |
| 已验证 | 内容已完成，检查或测试已通过，尚未推送 |
| 已提交 | 原子提交已推送，等待计划进度回写 |
| 已完成 | 原子提交和计划更新提交都已推送 |
| 阻塞 | 缺少环境、凭据、决策或外部服务，需要用户协助 |

任意时刻最多只能有一个“进行中”项。

## 4. 设计项流程

### 4.1 开始条件

- 从`docs/implementation-plan.md`选择下一个满足前置条件的设计项；
- 将状态改为“进行中”；
- 确认本次只修改计划允许的文档路径；
- 确认不包含编码和运行环境搭建。

### 4.2 设计文档最低内容

详细设计文件统一放在`docs/design/`，至少包括：

1. 背景与目标；
2. 本次范围；
3. 明确非目标；
4. 组件职责和调用关系；
5. 输入输出与接口契约；
6. 数据模型和状态变化；
7. 正常流程和异常流程；
8. 幂等、回退和兼容策略；
9. 安全与证据追踪要求；
10. 单元、集成和端到端测试设计；
11. 所需运行组件和环境检查；
12. 风险、替代方案和未决问题。

### 4.3 设计反思

设计完成后必须增加“设计反思”章节，回答：

- 是否存在更小、更直接的实现方式？
- 是否引入了当前阶段不需要的组件？
- 是否与`docs/architecture.md`冲突？
- 失败时是否能恢复，数据是否会被污染？
- AI是否越过了审核、校验或真实推理边界？
- 端到端测试是否真的能证明本次能力？
- 是否存在未经确认的假设？

### 4.4 设计检查

- 检查范围、接口、数据、异常和测试是否完整；
- 检查计划项与设计文件编号一致；
- 检查文档链接、Mermaid和代码围栏；
- 使用`git diff --check`检查格式问题；
- 使用`git diff --name-only`确认没有越界文件；
- 将检查结论写入设计文档。

设计检查通过后，按以下格式提交并推送：

```text
design(Pxx-D): 完成某模块详细设计
```

推送成功后更新计划表，设计项才能标记为“已完成”。

## 5. 实现项流程

### 5.1 开始条件

- 对应设计项已完成并推送；
- 设计中的未决问题已经关闭；
- 所需运行时、依赖和凭据已确认可用；
- 当前项允许修改的路径已经在计划表中明确。

### 5.2 编码规则

- 严格按照已批准设计实现；
- 不顺带创建后续模块骨架或占位实现；
- 新发现需求写入计划，不在当前提交扩展；
- AI模拟输入必须使用统一接口并标记来源；
- 正式事实只能经过审核和校验进入；
- 推理结果必须由真实规则或推理引擎产生。

### 5.3 测试要求

每个实现项至少执行：

1. 静态或语法检查；
2. 当前模块单元测试；
3. 与直接依赖组件的集成测试；
4. 计划表规定的端到端验证；
5. 失败路径或错误输入测试。

测试命令、环境、输入、预期和实际结果必须记录到计划表或本次设计文档的验证记录中。

### 5.4 代码检视

提交前必须检查：

- 实现是否符合设计和验收条件；
- 是否修改了允许路径之外的文件；
- 是否存在未处理异常、重复执行或数据污染风险；
- 是否泄露密钥、内部地址或模拟数据误标；
- 测试是否覆盖主要成功与失败路径；
- 是否混入调试文件、生成物和无关格式化；
- `git diff --check`和`git status --short`是否符合预期。

代码检视结论必须形成简短记录。

### 5.5 提交、推送和计划回写

实现提交格式：

```text
feat(Pxx-I): 实现某个原子能力
fix(Pxx-I): 修复某个原子缺陷
test(Pxx-I): 补充某个原子验证
```

执行顺序：

```text
完成实现
→ 测试和端到端验证
→ 代码检视
→ 原子提交
→ 推送GitHub
→ 更新计划状态、提交号和验证证据
→ 提交计划更新
→ 再次推送GitHub
→ 开始下一项
```

## 6. 端到端验证定义

端到端验证不能只检查函数返回值，必须从本次能力的公开输入边界开始，经过真实目标组件，到达公开输出边界。

示例：

```text
输入TTL本体
→ pySHACL真实校验
→ Fuseki真实入库
→ Jena真实推理
→ API真实查询
→ 验证候选无人机及其推理依据
```

如果目标组件在当前环境缺失，计划项标记为“阻塞”并向用户求助，不能用假实现替代后宣称端到端通过。

## 7. 越界控制

以下行为视为越界：

- 在本体项中顺带实现前端；
- 在存储项中顺带接入AI；
- 在API项中顺带修改未批准的本体语义；
- 为未来功能提前创建未验证的框架代码；
- 为通过测试而绕过真实开源组件；
- 将多个计划项合并为一个提交。

越界需求应新增或调整后续计划，并经用户确认后执行。

## 8. 环境阻塞处理

缺少以下内容时应停止并说明：

- Docker或对应容器运行环境；
- Java、Python、Node.js等设计指定运行时；
- Fuseki、Tika、推理器或浏览器运行能力；
- AI服务配置和凭据；
- GitHub有效认证；
- 执行端到端测试所需的端口、文件权限或网络权限。

求助信息应包含：当前计划项、已经完成的检查、缺少的组件、建议安装方式和恢复后需要执行的验证命令。
