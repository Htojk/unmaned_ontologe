## 3. 架构反思

### 3.1 是否存在更小的实现方式

完整目标架构包含前端、后端、Fuseki、Tika、AI和推理组件，但实施计划没有一次性搭建全部系统，而是按“环境—本体—存储—校验—推理—证据—模拟—抽取—审核—前端—集成”拆成独立纵向能力。

结论：总体架构完整，实施方式已收敛为更小原子项。

### 3.2 是否过早引入组件

- OCR不是第一版必选服务，仅扫描件场景启用；
- HermiT主要用于建模期检查，不在第一版部署独立运行服务；
- 第一版不引入消息队列、对象存储、多Agent框架和生产级权限系统；
- AI适配器在确定性离线场景之后实现，避免AI成为基础闭环前置条件。

结论：组件引入顺序合理，未要求提前建设非必要基础设施。

### 3.3 AI边界是否清晰

架构和准则共同规定：

- AI只模拟输入和审核动作；
- AI输出必须经过结构化校验；
- AI不能直接写正式事实图或推理图；
- 校验、存储、推理和查询由真实组件完成；
- AI不可用时使用fixture完成相同核心断言。

结论：AI不能替代真实系统处理或伪造演示结论。

### 3.4 是否能够失败恢复

架构定义了候选图、正式事实图、推理图和证据图隔离；推理图可重建；AI失败可降级；校验失败返回审核；推理失败不替换旧结果。

结论：主要失败路径和恢复边界已经覆盖，具体事务策略由对应详细设计项继续细化。
